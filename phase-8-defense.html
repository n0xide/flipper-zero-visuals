Phase 8 — Defense, audit, hardening

The blue-team payoff. Every attack from Phases 2-7 turned into a defense recommendation. If you can't write the hardening section, you don't understand the attack.

Posture: defense is decision support, not security theatre. The matrix below maps each attack to (a) detection — how would a defender notice? — and (b) hardening — what control closes it? Each row also carries a cost tag so you can argue for the controls in budget terms when the engagement lands at the board level. This is the document you hand a client at the end of an audit, scaffolded.

2. Detection layers — who catches what

Detection isn't a single thing. Different attacks surface in different telemetry; pretending one product catches everything is how attacks slip through.

LayerWhat it seesCatchesMissesTooling
Physical / CCTV Bodies near hardware, unattended workstations, "stranger in server room" Most physical-access attacks at the entry stage (Bad USB drop, hardware tampering) Insiders, after-hours when monitoring is light IP cameras, badge logs, alarm systems, signed-in-visitor logs
RF monitoring Spurious transmissions, anomalous WiFi APs, deauth storms Rogue APs (evil twin), sub-GHz brute-force attempts, jamming Passive captures, off-frequency / SDR work Kismet, Wireshark+monitor, Bastille SAFE, AirEye, simple SDR + script
Endpoint EDR Process tree, file writes, registry changes, network connections from each host Bad USB Stage 2 payloads, persistence, lateral movement Hardware-only attacks (SWD/SPI flash dump) that never touch the OS CrowdStrike Falcon, Microsoft Defender for Endpoint, SentinelOne, Elastic
Network / NTA Flow records, TLS metadata, DNS lookups, unusual destinations C2 callback, DNS tunneling, data exfil patterns Encrypted-content payloads, traffic to allowed cloud SaaS as C2 Zeek, Suricata, Darktrace, Palo Alto Cortex XDR, Vectra
Access logs Door reads, building entries, time stamps Card cloning when the clone is used (same card "in" twice without an "out") First use of a clone before any anomaly accumulates Access-control system (AMAG, Lenel, etc.), correlation in SIEM
USB telemetry Device-add events, VID/PID, descriptor strings, keystroke cadence Bad USB drops, USB-storage exfil, HID injection floods Slow-typing Bad USB tuned to look human EDR USB module, Windows Event ID 6416, Linux udev + logging
Hardware tamper Case-opened sensors, SOIC clip insertion, hardware-attestation failure Physical hardware attacks (chip-off, glitch, SWD probe) Everything if no sensor is fitted (most consumer hardware) Tamper-evident seals, case-intrusion switches, secure boot + attestation reports
The principle: defense in depth means the attacker has to evade every layer. One camera + one EDR + one network sensor → the attacker has to be invisible to all three. Most successful attacks succeed because one of the three layers wasn't actually deployed despite the budget line item; audits should verify deployment, not just procurement.

3. Audit deliverable checklist — what a client gets

This is a template, not a script. Strike anything that doesn't apply to the engagement scope. Add sections as the engagement uncovers new surface. The final report should be terse enough that a non-technical decision-maker reads it cover-to-cover.

Section A — Executive summary (1 page)
  • Scope: building / floor / system list, dates of testing
  • Authorization: who signed, when, what was in/out of scope
  • Top 3 findings: severity + business impact, in plain English
  • Top 3 recommendations: budget tag, owner, timeline
  • Residual risk after recommendations implemented
Section B — RF / physical-access surface
  • Sub-GHz: all fixed-code remotes inventoried; rolling-code deployments verified non-replayable
  • LF RFID: card formats identified; any 26-bit HID Prox / EM4100 flagged for replacement
  • HF NFC: MIFARE Classic deployments flagged for migration; DESFire / Plus-SL3 deployments validated
  • iButton: any building-access iButton flagged for second-factor or replacement
  • IR: not a security control — note any IR-only controls (door bells, remote shutdowns) for awareness
Section C — Endpoint / Bad USB surface
  • USB port policy verified (allowlist / blocklist / disabled in non-essential ports)
  • HID-injection EDR rules verified (typed/created test rule and observed alert)
  • PowerShell Constrained Language Mode + AMSI status across endpoint fleet
  • WDAC / AppLocker deployment depth — sample of typical workstation tested
  • Egress filtering for typical workstation: DNS-tunnel test, common C2 destinations
  • User-training program: phishing + USB-drop result rates from last drill
Section D — Wireless surface
  • WiFi: WPA3 + PMF mandatory; any WPA2-only with deauth-vulnerable PMF-off configurations
  • PSK strength — sample audit of staff-shared passphrases (with consent / in scope)
  • Guest WiFi: OWE or strong isolation from corporate; no path between
  • BLE: corporate-issued mobile policy for BLE address randomization where applicable
  • NRF24 KB/mouse receivers — any in use? Replace with Bluetooth or wired.
Section E — Hardware / IoT surface
  • Internet-facing IoT inventoried; default credentials verified changed; firmware update path validated
  • For high-value IoT (cameras, building automation): SPI-flash dump performed, firmware analyzed for hardcoded secrets
  • UART debug headers present on production hardware — disable in next revision
  • Cortex-M / ARM RDP level on critical embedded controllers verified at level 1 or 2
  • Vendor disclosure log: any novel vulns found, status with vendor
Section F — Detection coverage
  • SIEM rules deployed for each Phase-2-7 attack class — coverage matrix attached
  • Incident-response runbook updated for each attack class
  • Tabletop exercise scheduled within 90 days using findings from this audit
Section G — Appendix
  • Tool list + versions used during the engagement
  • Network captures + Flipper save files (delivered separately, encrypted)
  • Findings register: every vuln found with CVSS, impact, suggested fix
  • Engagement logs: every test, timestamp, scope justification
  • Sign-off: client acknowledgement of receipt + agreed remediation owners

4. Priority — what to fix first

The matrix in §1 lists every attack and its hardening. Real engagements never have budget for all of them at once. This is the ordering I'd recommend across most general-purpose enterprises.

#ActionWhy first / why laterCost
1 Enable PMF + migrate to WPA3 on all WiFi Single config change kills the deauth-class attack surface entirely. Most APs support it; the holdouts are the ones you want to replace anyway. cheap
2 Disable PowerShell v2; enforce CLM + AMSI logging Catches the bulk of Bad USB / phishing payload chains. Already shipped in Windows; turn it on. cheap
3 Replace any MIFARE Classic / EM4100 / HID Prox 26-bit deployments Clonable in 30 seconds with $50 of gear. The breaches happen; replacement to DESFire / Seos is a few hundred dollars per door. med
4 USB device-control policy (allowlist VID/PID, alert on new HID) Bad USB defense onion ring 2 — cheap and high-yield. EDR usually already has the capability. cheap
5 NRF24 keyboard/mouse receivers → Bluetooth or wired MouseJack is a 10-year-old attack; cheap to fix per workstation. med
6 Egress filtering + DNS monitoring for unusual destinations Catches Stage-2 callbacks. Many shops already pay for the tooling but underuse it. med
7 WDAC / app allowlisting on critical hosts (DCs, servers, exec workstations) The "expensive layer" of the onion. Operationally heavy; do critical machines first, not everywhere. expensive
8 Hardware tamper evidence on critical IoT / embedded Only the high-value targets (door controllers, badge readers, KVMs). Don't try to do it everywhere. expensive
9 Migrate iButton-controlled doors to PIN + iButton, or to RFID iButton alone is trivially cloned. PIN compensates; full replacement is better but slower. med
10 Tabletop exercise using the audit findings within 90 days The audit becomes useful only when the response team can talk through it. cheap
Anti-pattern to avoid: "we'll do them all next quarter." That promise rarely survives the next quarter's priorities. Pick 3 from this list, fund them, set names against them. The remaining 7 go into a quarterly roadmap with explicit owners.
