Phase 3 — Card Technologies (LF RFID + HF NFC)

Interactive visualizations for the two contactless families. The dividing line: pre-2005 systems = LF, trivial. Post-2010 + budget = HF DESFire, unclonable. The middle ground (MIFARE Classic) is where the interesting attacks live.

Legality boundary — who owns the card / who owns the system?
  • Reading + cloning your own cards (your house key fob, your office prox card with HR sign-off, blank T5577/Magic Mifare you bought) — legal. This is the primary lab workflow below.
  • Reading a third-party card (someone's wallet brushed against your reader, a hotel key, a transit card you didn't buy) — varies by jurisdiction. In the US, unauthorized access to a system that authenticates via the card can fall under the CFAA even if the read itself is passive. Get written consent for any card that isn't yours.
  • EMV (credit/debit) reads — Flipper can read PAN + expiry on contactless cards. Even passive demonstration on a card you don't own can be charged as access-device fraud (US 18 USC §1029) in some districts. Practice on your own card only.
  • Cloning to attack a building access system (replaying a coworker's badge, fabricating a manager's card) is unauthorized access — felony in most jurisdictions. The visual below teaches it because the defense recommendations follow from understanding it, not because the technique is permitted in the wild.
Same rule as Phase 2: teach the technique fully, point it only at hardware you own or are authorized to test.

2. EM4100 — 64-bit packet, every bit accounted for

The format that singlehandedly justified migration to HF. Hover any bit to see what it does. Note: no crypto. No challenge-response. Tag broadcasts this every read.

64 bits = 9 header + 40 data + 14 parity + 1 stop
Header (9 × 1s)
Facility code (8 bits)
Card number (32 bits)
Row + column parity (14 bits)
Stop bit (1)
Reading this: the 40 data bits are organized as 10 rows × 4 bits, each row followed by an even-parity bit. The 4 final column-parity bits cover the columns. None of this is cryptographic — it's just error detection. Anyone with a 125 kHz coil and a few hundred lines of code can clone this.
Why parity, but no crypto?

EM4100 was designed in the 1990s for industrial item-tracking. The threat model was "did the noisy industrial radio environment garble the read?" not "is someone trying to clone this." Parity catches single-bit corruption from the analog channel — that's all it was meant to do. The standard never went back to add crypto because the cost (chip silicon, fab change) outweighed the perceived need until a long-after-the-fact migration to HF.

3. NFC family tree — what's broken, what isn't

"Is this clonable?" depends entirely on which family. Don't paint all NFC with one brush.

Caveats on the "SECURE" box:
  • MIFARE Plus ships in three Security Levels. SL1 is backwards-compatible with Crypto1 and clonable like a Classic; SL3 is AES-only and unclonable. A "Plus" card by name tells you nothing — the deployment chooses the level.
  • Ultralight C uses 3DES with a 16-byte key; NTAG21x uses an ECDSA originality signature (NXP curve, secp128r1) + password protection (PWD/PACK) + a one-way mirrored counter. Different chips, similar form factor — emulation only fools naive readers, signature validation catches clones. (CMAC + SUN belong to NTAG 424 DNA, not 213/215/216.)
  • "Magic" Mifare: Gen1a = undocumented "backdoor" command pair for block-0 write; Gen2 / CUID = block 0 rewritable through normal MIFARE write after auth; Gen3 = dedicated APDU command changes UID (block 0 updates automatically, ATQA/SAK fixed); Gen4 = programmable ATQA/SAK/config (most flexible spoofing).

5. The Crypto1 attack ladder — when to use which

Five distinct attacks (matrix below). They're not alternatives — they're rungs of a ladder. Try cheap things first; escalate only on failure.

Pick the right rung — decision matrix

AttackHave card?Have reader?Have ≥1 key?Works against hardened?When to reach for it
1. Dictionary yes — — partial Always first. ~70% of real-world cards crack on default keys alone.
2. Nested yes — ≥1 no One key bootstraps the rest. Cheap, fast, your default after dictionary.
3. Hardnested yes — ≥1 yes MIFARE Classic EV1 / "hardened" gen2 cards that defeated plain nested. CPU-heavy.
4. Darkside yes no none no The blackbox case: a card and nothing else. Forces failed-auth attempts, exploits parity-bit leak (Garcia et al., 2009). Slow (minutes). Then chain into nested.
5. mfkey32 no yes — yes You can present the Flipper to the legitimate reader, but the card itself isn't available. Reader nonces leak the key.

Stock Flipper firmware ships dictionary + nested + mfkey32. Hardnested and darkside live in alternate firmwares (Momentum, Xtreme) or as offline desktop tools (libnfc mfoc for nested, mfcuk for darkside).

6. EMV (credit/debit) — what the Flipper can and can't do

The most common misconception in tech-press articles. Let's draw the line precisely.

Why this matters: "Flipper Zero steals credit cards" is a misleading headline. The Flipper performs the same APDU exchange any terminal does. The difference is the Flipper never sees the issuer's private signing key. Without that key, the cardholder cryptogram cannot be reproduced, so no terminal will accept a transaction. Caveat: compromised terminals and relay attacks (paired devices bridging a real card and a real reader live) are real — but require coordinated infrastructure beyond a solo Flipper.
